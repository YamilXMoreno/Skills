# Contributor guide — building a HiL-Bench task, step by step

This is the hands-on walkthrough for authoring an outcome_ladybug (HiL-Bench) task with the
`hilbench-pipeline` skill. It is task-oriented: for each stage you get the command to run,
what to review before moving on, and how to read the result. For a precise per-command
breakdown (inputs, the script/reference each command runs, the exact checks it performs, what
it writes, and every result sentinel) see [`COMMAND_REFERENCE.md`](./COMMAND_REFERENCE.md).
For architecture and internals see [`README.md`](./README.md); for the sandbox/platform
picture see [`../SANDBOX_SETUP.md`](../SANDBOX_SETUP.md).

> **Already have blockers? This isn't your guide.** This walkthrough is for *authoring* a task
> from scratch. If the task already ships with a blocker set — a full `deliverables/` folder
> (synthetically generated by the pipeline or authored by an earlier contributor) plus async eval
> results — you're reviewing, not authoring: use [`REVIEWER_GUIDE.md`](./REVIEWER_GUIDE.md) instead.

**Golden rule:** every command STOPS for your review. Nothing auto-advances. You decide when
to run the next stage. When something fails a gate, fix it and re-run that stage — do not
push forward.

> **You are the reviewer — the agent is not.** Every prompt-driven step (registry authoring,
> injection, the model evals, the patch-content validator, the Attempter Check solves, and any
> triage/repair output) requires your own critical analysis. These outputs are **not 100%
> accurate**: the agent can and does make mistakes, and it is most likely to be wrong when it
> **makes assumptions about the project's goals** — inferring intent, filling gaps with plausible-
> but-wrong defaults, or declaring `PASS`/`TRUE` on reasoning that doesn't actually hold. Read
> every response carefully, verify its claims against the actual code, tests, and specs, and do
> not treat a green verdict as ground truth. The scripted checks (`task_checker.py` runs) are
> deterministic, but any step where the agent *reasons or writes prose* is advisory and must be
> independently confirmed by you before you proceed.

> **Use Auto mode, not Bypass.** In Claude Code, **Auto mode** is the recommended way to run the
> `/hilbench-*` commands: it approves safe actions and pauses for risky ones (destructive shell,
> `docker`, overwrites). **Do NOT use "Bypass permissions".** It runs everything with no sign-off.
>
> **Always read a fix before you run the check again.** In Auto mode, the agent can change your
> files on its own. So when a check FAILs and the agent tries to fix it, it may edit your answer
> key (the registry, the modified specs, or the golden patch) without asking you first. Running the
> check again only tells you if it passes now. It does not tell you if the fix was correct. So look
> at what the agent changed before you run the check again. If you want the agent to ask you before
> a specific fix, switch to Manual for that edit, then switch back to Auto. The scripts always give
> the same result, but anything the agent writes or decides can be wrong, so you must check it.

## How and where to run things

There are two surfaces, and it matters which one you use:

- **The chat/LLM window** (the Cursor/Claude Code agent chat). This is where you run the
  **`/hilbench-*` slash commands**. Type the command (e.g. `/hilbench-provision`) into the
  chat prompt and send it. The agent reads the skill, runs the underlying scripts for you,
  and prints the result. In this guide these are marked **[chat]**.
  - If your setup doesn't expose the slash command yet, you can instead just tell the agent
    in plain language, e.g. "run the hilbench-provision command / stage", and it will follow
    the same command file.
- **The terminal** (the web terminal / shell in the sandbox). This is where you run plain
  **shell commands** yourself: `source deliverables/.hilbench_env`, `docker exec ...`,
  `docker start ...`, inspecting files, etc. In this guide these are marked **[terminal]**
  and shown in shell code blocks.

Rule of thumb: anything starting with `/hilbench-` → **chat window**. Anything that looks
like a normal shell command (`source`, `docker`, `git`, `ls`) → **terminal**.

You can keep both open side by side: run a slash command in chat, then use the terminal to
inspect what it produced (diffs, files, logs) during the review pauses.

> Note: the slash commands ultimately just invoke the scripts in `scripts/`. If you ever
> want to run a step manually, you can call the script directly in the **terminal**, e.g.
> `bash ~/.claude/skills/hilbench-pipeline/scripts/provision_repo.sh`. The chat commands are
> the recommended path because they also handle path resolution, the review gates, and the
> fresh-subagent solve for the checks.

### Exploring the container contents

Once `/hilbench-provision` has started the container (and you've `source`d
`deliverables/.hilbench_env` so `$HILBENCH_CONTAINER` / `$HILBENCH_REPO` are set — default
`hilbench_task` and `/app`), you can look inside the real repo in two ways.

**Option 1 — attach an interactive shell (recommended).** Run this **in your own terminal**
(not through the chat agent — it cannot host an interactive TTY). `-w` starts you in the repo:

```bash
docker exec -it -w "$HILBENCH_REPO" "$HILBENCH_CONTAINER" bash
```

That drops you into a shell at `/app`. Type `exit` to leave — the container keeps running
(it's a `sleep infinity` container, so detaching your session does not stop it).

If you'd rather browse from your host with your own editor, snapshot the tree out:

```bash
docker cp "$HILBENCH_CONTAINER":"$HILBENCH_REPO" /tmp/hilbench-app   # one-way copy to host
```

Note: `docker cp` is a **one-way** copy — edits under `/tmp/hilbench-app` do NOT affect the
container. And do not hand-edit the container's `/app` for the pipeline anyway: the later
stages reset git in it on every run, so any manual changes get discarded. Let the commands
author the diffs (they capture them via `git diff` inside the container).

**Option 2 — ask the chat agent to run read-only commands for you.** Tell it what you want to
see and it will `docker exec` it. For example:

```bash
docker exec -w "$HILBENCH_REPO" "$HILBENCH_CONTAINER" git log --oneline -5
docker exec -w "$HILBENCH_REPO" "$HILBENCH_CONTAINER" ls persistence
docker exec -w "$HILBENCH_REPO" "$HILBENCH_CONTAINER" cat model/album.go
```

(Those example paths are illustrative — substitute the files/dirs your task actually touches.)

## 0. Mental model (read once)

You are turning a solved SWE task into an *obstructed* task: you inject "blockers"
(ambiguities / missing values / contradictions) that a good agent cannot resolve without
asking, then you prove (a) the blockers are not guessable and (b) the task is still solvable
once the answers (resolutions) are known.

Two places your work lives:

- **Sandbox** (durable): your inputs in `task_files/`, everything you author in
  `deliverables/`, the skill, and the model. This survives.
- **Container** (disposable): a SWEAP Docker image holding the real repo at `/app` plus the
  language toolchain. You provision it up front; it can be stopped/rebuilt anytime.

You (the human) own the answer key. The model helps you draft and check, but the blocker
*resolutions* are your responsibility to get right.

## 1. Prerequisites

- The `hilbench-pipeline` package is installed (`bash install.sh`). It is self-contained —
  the Harbor agentic-check skills (`input_validation` + `evaluate_task`) are bundled, so no
  separate `hilbench-evaluation` install is required.
- Docker is available in the sandbox.
- `task_files/` contains `task_info.txt`, `test_patch.diff`, `golden_patch.diff`, and
  `task_checker.py`. `task_info.txt` must include the docker pull command / image, the
  `base_commit_hash`, the `instance_id`, and the required blocker distribution.

If a required input is missing you'll see `REQUIRED_INPUT_FILE_MISSING` — stop and get the
file before continuing.

## 2. Stage 0 — Provision the repo/container

In the **chat window**, run:

```
/hilbench-provision
```

What it does: resolves the image source once (a registry ref is pulled; a URL is fetched and
either `docker build`-ed from a Dockerfile, `docker load`-ed from a tarball, or extracted into a
pull), starts a long-lived container, verifies the repo's `HEAD` matches `base_commit_hash`
— **auto-checking out the base commit if the image didn't ship at it** — and writes
`deliverables/.hilbench_env`.

> Tip: add `--validate-original` to chain Stage 0.5 immediately after a successful provision
> (`/hilbench-provision --validate-original`). By default the two are separate stages.

**Before moving on:**
- Confirm the chat prints `PROVISION_OK`.
- Then, in the **terminal**, load the task context so later stages know the container,
  instance id, and repo:

```bash
source deliverables/.hilbench_env
```

**If you see `PARENT_COMMIT_MISMATCH`:** STOP. Every patch is written against the parent
commit; a wrong baseline makes all your diffs invalid. Because auto-checkout is now the
default, this only fires when the base commit **could not be checked out** — it isn't in the
repo's history (re-check the image tag / that the clone includes the base commit), or you ran
with `--no-checkout`. It is no longer a "the image shipped the wrong commit" signal; that case
is fixed automatically.

**If you see `UNRECOGNIZED_IMAGE_SOURCE`:** STOP. The image URL in `task_info.txt` was fetched
but couldn't be classified as a Dockerfile, tarball, or pull command — check the source URL, or
pass `--image` with a local tag you built yourself.

**If you see `IMAGE_BUILD_FAILED` (or `IMAGE_LOAD_FAILED`):** the source was recognized (a
Dockerfile / tarball) but the `docker build` / `docker load` itself failed — an environment or
Dockerfile problem, not a bad URL. The build path already retries with the legacy builder
(`DOCKER_BUILDKIT=0`) to survive rootless-Docker/buildx breakage (e.g. it can't chown
`~/.docker/buildx/instances`), so both builders failed here. Read the build error; if it's a
transient/builder issue, pre-build the image yourself and pass `--image` with that local tag.

> Tip: to pause work, run `/hilbench-teardown --stop-only` in **chat** (keeps the container
> for a fast `docker start <name>` in the **terminal**). To fully clean up, `/hilbench-teardown`
> (image + your deliverables are kept). Re-run `/hilbench-provision` before any check/eval stage.

> **Where does the docker/git setup happen?** Stages 0 and 0.5 automate what the instructions
> call the *Codebase Editing Workflow* — acquire the image (pull/build/load), run container,
> open `/app`, verify (and if needed check out) the parent commit (`git rev-parse HEAD`), and
> validate the original patches. You no longer do those docker/git commands by hand; the
> commands do them for you (via `scripts/`).

## 2.5. Stage 0.5 — Validate the original patches

Before you invest any effort in injection, confirm the baseline SWE task is sound: the
ORIGINAL golden + test patches must apply on the verified parent commit and show a clean
FAIL→PASS. (This is step 5 of the Codebase Editing Workflow, "Validate the Original Patches".)

In the **chat window**:

```
/hilbench-validate-original
```

For the full FAIL→PASS check the command first builds a throwaway JSON list of the tests in
the original test patch (`$TASK_FILES/relevant_tests_original.txt`), then runs
`task_checker.py` on the original patches. If it can't build that list or the instance id is
unknown, it falls back to just confirming both patches apply.

**Before moving on:**
- `ORIGINAL_PATCHES_OK` (F2P) or `ORIGINAL_APPLY_OK` (apply-only) → baseline is sound; proceed.
- `ORIGINAL_TEST_PATCH_APPLY_FAIL` / `ORIGINAL_GOLDEN_PATCH_APPLY_FAIL` → the originals don't
  apply on this commit; you're almost certainly on the wrong baseline (re-check provisioning —
  did it print `PARENT_COMMIT_MISMATCH`?) or the input patches are corrupted. **Do not start
  injection.**
- `ORIGINAL_F2P_FAIL` → the original patches don't cleanly go FAIL→PASS for the tests; verify
  the baseline before proceeding.

## 3. Stage 1 — Author the blocker registry

Pick **one** of three ways to produce `blocker_registry.json` + `blocker_registry.md`. The
slash commands run in the **chat window**; the manual path is you editing files (in the
editor or **terminal**):

| You want… | Run |
|-----------|-----|
| The model to propose everything in one shot from the originals | `/hilbench-registry-generate` **[chat]** |
| To decide + lock the scenario/config up front, then build the registry (best when the originals likely can't enforce your whole distribution — i.e. Scenario 2) | `/hilbench-plan` then `/hilbench-registry` **[chat]** |
| To write/adjust the registry by hand (or your own prompt) | edit `deliverables/blocker_registry.json` directly (editor/terminal) |

> **When to use the two-step plan.** Don't pick it just because narrow tests exist — narrow
> tests point at the *simple* case (Scenario 1). The plan step earns its keep in the opposite
> situation: when the original tests are broad, or you can't tell whether they enforce **every**
> required blocker (i.e. Scenario 2 is likely). It runs the scenario feasibility analysis and
> writes `plan.md`, so the later inject step reads the decision instead of re-deriving it. For a
> clear Scenario 1, the one-shot generator is fine. See Stage 2 for what the scenario actually
> controls.

Whichever you choose, **review the resolutions yourself** — they are the answer key. Check
that each blocker is realistic, critical, objective, and not guessable, and that the counts
match the required distribution.

### Gate: validate the registry

In the **chat window**:

```
/hilbench-validate-registry
```

This runs on **every** authoring path (generator, numbered, or manual). It checks schema/shape
first, then the static-quality evals — the model evals run **verbatim** from
`references/eval-prompts.md` (distribution, self-reference, objectivity, independence, type,
descriptions, trigger-question relevance, and criticality **Test B/C only** — necessity is
deferred to the artifacts gate), plus the guessability and description/type-coherence linters.

**Before moving on:** you must see `REGISTRY GATE: PASS`. On `FAIL`, the output names the
failing check and blocker — fix that blocker (re-author or hand-edit) and re-run this gate.
Do not inject until it passes.

## 4. Stage 2 — Inject the blockers

### First: which Scenario are you in?

The Scenario controls one thing — **whether the `/hilbench-inject` step has to write new
patches**:

- **Scenario 2 (normal).** The originals can't enforce your whole blocker set, so you author
  obstructed patches too: `test_patch_obstructed.diff`, `golden_patch_obstructed.diff`, and
  optionally `setup_patch.diff`.
- **Scenario 1 (rare).** The original test patch *already* has a narrow assertion enforcing
  **every** required blocker, so you only rewrite the text and reuse the originals unchanged.

The Scenario 1 bar is high — one blocker without an enforcing test in `test_patch.diff` puts
you in Scenario 2 — so most tasks are Scenario 2.

**You normally pass nothing.** Inject picks the scenario from `--scenario` if given, else
`plan.md`, else it decides and prints its call for you to confirm before injecting. So
`--scenario` is just an override; the only common use is forcing `--scenario 2` when you don't
trust an auto-detected "Scenario 1."

```
/hilbench-inject --scenario 2   # only to override a wrong auto-detected Scenario 1
```

### Run it

In the **chat window**:

```
/hilbench-inject
```

Writes the obstructed task text: `modified_problem_statement.txt`,
`modified_requirements.txt`, `modified_public_interfaces.txt`. On **Scenario 2** it also
produces `setup_patch.diff` (if needed), `test_patch_obstructed.diff`, and
`golden_patch_obstructed.diff` by editing the real repo in the container and capturing diffs.
On **Scenario 1** it reuses the original patches (no logic changes). Finally it writes
`relevant_tests.txt` (step 06) — the JSON list of **all** tests in the authoritative test
patch (the obstructed one on Scenario 2), each string discoverable verbatim.

**Before moving on:** open the modified text and confirm it does NOT leak any resolution
value, contains no self-referential wording ("blocker", "removed", "registry", etc.), and
still leaves the intended ambiguity in place. For Scenario 2, skim the obstructed diffs.

### Gate: validate the artifacts

In the **chat window**:

```
/hilbench-validate-artifacts
```

Two parts. **Part A** is the step-08 patch-content validator + alignment map: confirms every
relevant test is in the test patch, each resolution has an enforcing test, the golden patch
implements the full spec + resolutions, and test/golden patches don't overlap. **Part B** runs
the patch-dependent model evals **verbatim** from `references/eval-prompts.md` now that the
patches exist: `eval_request` (#12 — the spec leaks no resolution / embeds no solution) and
`eval_blocker_critical_implementation` **Test A / necessity** (#7 — each resolution is actually
required to pass a relevant test). Check 1 later confirms eval-7 Test B empirically.

**Before moving on:** you must see `ARTIFACTS GATE: PASS` (with the alignment map). On
`FAIL`, prefer fixing the golden patch or adding the missing enforcing test — never weaken
the tests. See [`references/optional-repair.md`](./references/optional-repair.md).

## 4.5. Stage 2.5 — Validate the obstructed patches (dynamic FAIL→PASS)

GATE 2 above only validated your obstructed golden **statically** (an LLM reads the diffs). Before
spending Attempter-Check budget, confirm the authored golden actually **executes** cleanly. In the
**chat window** (container must be running):

```
/hilbench-validate-obstructed
```

This bakes `setup_patch.diff` as the baseline, then runs `task_checker.py` with
`test_patch_obstructed.diff` + your **authored** `golden_patch_obstructed.diff` — confirming the
obstructed test fails on baseline+setup and the golden makes it pass. It grades *your* golden (the
Attempter Checks grade an agent-generated patch instead), so it's the direct "apply setup → golden →
run tests" check.

**Before moving on:** you must see `OBSTRUCTED_PATCHES_OK`. On `OBSTRUCTED_F2P_FAIL`, align the
golden/tests/spec (never weaken tests) and re-run GATE 2 + this check; on an apply-fail, regenerate
the offending diff (see [`references/07-patch-outputs.md`](./references/07-patch-outputs.md)).

## 5. Stage 3 — Attempter Checks (optional fast pre-screen, inside the container)

These are a fast, local, single-model smoke test of your blockers, run inside the container.
They are **optional but recommended**: run them to catch an obviously guessable or unsolvable
task cheaply *before* you spend the required Harbor evaluation (Stage 4) budget. They are NOT
a substitute for Stage 4 — a single-model local PASS can still fail Harbor's multi-model
threshold, so they never replace the required grade. Each is three steps and pauses for your
review before scoring.

### Check 1 — are the blockers guessable?

In the **chat window**:

```
/hilbench-check1
```

1. Prep: resets the repo to a clean baseline and scrubs the answer-key files from the
   container.
2. Solve: a fresh agent (clean context) gets ONLY the modified problem statement /
   requirements / interfaces — **no** resolutions, tests, or golden patch — and edits the repo.
3. **Review pause:** the chat stops here. In the **terminal**, inspect the agent's diff
   before you let it score:

```bash
docker exec "$HILBENCH_CONTAINER" git -C "$HILBENCH_REPO" diff
```

4. Score (back in **chat**, tell the agent to continue / run the evaluate step).

**Read the result:**
- `CHECK_RESULT check1 PASS` → the agent could NOT pass the blocker tests without the
  answers ⇒ blockers are not guessable. Good — proceed to Check 2.
- `FAIL_GUESSABLE` → the agent solved it without resolutions ⇒ at least one blocker is
  inferable. Find which, redesign it (widen the plausible answers / remove leakage), re-run
  the registry gate, re-inject, and re-check.
- `CHECK_ERROR_PATCH` / `CHECK_ERROR_ENV` → a diff/environment problem, not a blocker signal.
  Fix the diff or env and re-run just the eval step.

### Check 2 — is the task solvable with the answers?

In the **chat window**:

```
/hilbench-check2
```

Same shape (including the **terminal** review pause to inspect the diff), but the solving
agent also receives the blocker **resolutions** (still no tests).

**Read the result:**
- `CHECK_RESULT check2 PASS` → the agent passed all relevant tests with the answers ⇒ the
  task is solvable. 
- `FAIL_UNSOLVABLE` → tests still fail even with the answers ⇒ the spec is underspecified or
  golden/tests are misaligned. Add the minimal missing detail to the right artifact
  (problem statement / requirements / interface / resolution) and regenerate the golden
  patch — do not loosen tests. If the gap is a **visible codebase prerequisite** (scaffolding,
  a dependency, or a config — not implementation logic and not a spec detail), the right home is
  `setup_patch.diff`, which must stay leak-free and must not touch tests. See
  [`references/optional-repair.md`](./references/optional-repair.md).

## 6. Stage 4 — Evaluation (REQUIRED)

This is the authoritative grade and is **required for every task**. It is heavier than and
stronger than the optional Attempter Checks above — multi-model Harbor SWE-Agent grading. All
are **chat window** slash commands.

- **Run this:** `/hilbench-evaluate-full` — input validation, then the full Harbor Check 1 +
  Check 2 (up to 12 SWE-Agent runs), then the overall `PASS`/`FAIL` verdict. This is the
  required gate.
- **Incremental re-runs:** `/hilbench-evaluate-full` is smart on re-run — it only re-executes the
  stages whose inputs changed since the last run (reusing the previous verdict for the rest), via a
  content-hash manifest at `deliverables/harbor/eval_manifest.json`. So after a fix you can just run
  it again and it won't repeat unaffected checks. Which edit re-triggers which stage is defined in
  [`references/evaluation-rerun.md`](./references/evaluation-rerun.md).
  - `/hilbench-evaluate-full --fresh` forces a complete re-grade. **Do one `--fresh` run as your
    final sign-off** before considering the task done.
- **Force one check:** `/hilbench-evaluate-check1` or `/hilbench-evaluate-check2` (6 runs each) run
  a single check unconditionally, bypassing the planner — handy when you know exactly which one you
  want.
- `/hilbench-validate-input` — standalone structural input validation (repo builds, patches
  apply, test coverage). Optional on its own, since `/hilbench-evaluate-full` runs it as its
  first step; use it for a quick buildable/applies-cleanly check before firing the heavier
  grading.

These delegate to the bundled `input_validation` / `evaluate_task` skills (shipped with this
package), so no separate install is required.

---

## 7. Stage 5 — Capture

When Checks pass and you're satisfied, click **Capture Files** so `deliverables/` is
harvested into the TextCollection step. The key outputs:

- `plan.md` (if used), `blocker_registry.json`, `blocker_registry.md`
- `modified_problem_statement.txt`, `modified_requirements.txt`,
  `modified_public_interfaces.txt`
- `relevant_tests.txt`, `test_patch_obstructed.diff`, `golden_patch_obstructed.diff`
- `setup_patch.diff` (Scenario 2, if any)
- Gate/validation results: `validate_artifacts_result.txt` (GATE 2 verdict + alignment map),
  `validate_obstructed_result.txt` (Stage 2.5 dynamic F2P verdict)
- Check artifacts for review: `check1_agent_patch.diff`, `check2_agent_patch.diff`,
  `*_after_stderr.log`
- Harbor evaluation feedback: `harbor/evaluation_feedback.md` (verdict + per-run summaries;
  written by `/hilbench-evaluate-full`) and the agent patches under `harbor/`

Then `/hilbench-teardown`.

## 8. Quick reference

The ordered command list is below. For what each command *does under the hood* — inputs, the
script/reference it runs, the checks it performs, outputs, and result sentinels — see
[`COMMAND_REFERENCE.md`](./COMMAND_REFERENCE.md).

All `/hilbench-*` lines are run in the **chat window**; the `source` / `docker` lines are run
in the **terminal**.

```
/hilbench-provision            # 0    start + verify container (--validate-original chains 0.5) [STOP]
source deliverables/.hilbench_env   # terminal, after provision succeeds
/hilbench-validate-original    # 0.5  original patches apply + F2P    [STOP]
/hilbench-registry-generate    # 1  author (one-shot)  ┐
/hilbench-plan                 # 1  author (step 01)   ├ pick one
/hilbench-registry             # 1  author (step 02)   ┘
/hilbench-validate-registry    # 1  GATE                              [STOP]
/hilbench-inject               # 2  modified text + relevant_tests (+ S2 patches) [STOP]
/hilbench-validate-artifacts   # 3  GATE (step 08 + alignment)        [STOP]
/hilbench-validate-obstructed  # 3.5 dynamic F2P on the AUTHORED obstructed patches [STOP]
/hilbench-check1               # 3  pre-screen: not guessable (optional, recommended) [STOP]
/hilbench-check2               # 3  pre-screen: solvable (optional, recommended)      [STOP]
/hilbench-evaluate-full        # 4  REQUIRED Harbor grade: input validation + Check 1 + Check 2 (heavy; incremental; --fresh forces full) [STOP]
/hilbench-evaluate-check1/2    # 4  force one check unconditionally (6 runs each)     [STOP]
/hilbench-teardown             #    stop/remove container
```

## 9. Troubleshooting

| Symptom | Meaning | Do |
|---------|---------|----|
| `REQUIRED_INPUT_FILE_MISSING` | an input/artifact isn't on disk | get the file; check you ran the prior stage |
| `UNRECOGNIZED_IMAGE_SOURCE` | the image URL in `task_info.txt` isn't a Dockerfile/tarball/pull command | check the source URL; pass `--image` with a built local tag |
| `IMAGE_BUILD_FAILED` / `IMAGE_LOAD_FAILED` | source recognized (Dockerfile/tarball) but build/load failed — build already retried with `DOCKER_BUILDKIT=0` | read the build error (env/Dockerfile issue); pre-build and pass `--image` with a local tag |
| `PARENT_COMMIT_MISMATCH` | base commit couldn't be auto-checked-out (not in history, or `--no-checkout`) | re-check image tag / that the clone includes the base commit |
| `ORIGINAL_TEST_PATCH_APPLY_FAIL` / `ORIGINAL_GOLDEN_PATCH_APPLY_FAIL` | original patch doesn't apply on parent | wrong baseline (re-provision; check it didn't report `PARENT_COMMIT_MISMATCH`) or corrupted input patch; don't inject |
| `ORIGINAL_F2P_FAIL` | original patches don't go FAIL→PASS | verify the baseline + your original tests list before injecting |
| `CONTAINER_NOT_RUNNING` | container stopped/removed | `/hilbench-provision` (or `docker start` if `--stop-only`) |
| `CONFIG_MISSING` / `PLAN_INVALID` | distribution or plan not readable | add the distribution to `task_info.txt`; re-run `/hilbench-plan` |
| `REGISTRY_INVALID` | schema/shape failure | fix the JSON structure; re-run `/hilbench-validate-registry` |
| `INSTANCE_ID_MISSING` | can't fetch run/parse scripts | pass `--instance-id`, or add `instance_id` to `task_info.txt` |
| `FAIL_GUESSABLE` | a blocker is inferable | redesign that blocker; re-validate; re-inject; re-check |
| `FAIL_UNSOLVABLE` | spec underspecified | add missing detail to the right artifact; regenerate golden |
