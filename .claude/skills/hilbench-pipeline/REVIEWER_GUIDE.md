# Task review guide — reviewing and fixing a HiL-Bench task

This is the workflow for reviewing a task that already has a blocker set in place and getting it
ready to submit. The task may have been generated by the pipeline or authored by an earlier
contributor — either way it arrives as a full `deliverables/` folder and you review it the same
way, then fix it if needed. There is no send-back.

> **If the task has no blockers yet, this isn't the right guide.** That's authoring — create the
> blockers using [`CONTRIBUTOR_GUIDE.md`](./CONTRIBUTOR_GUIDE.md), then it enters this review
> flow. For pipeline/command mechanics see also [`COMMAND_REFERENCE.md`](./COMMAND_REFERENCE.md).

> **You own the task once you submit it.** A prior PASS — self-graded by the generator or
> produced by an earlier reviewer — can hide a weak or guessable blocker, so double-check anything
> you change.

---

## What you receive

A `deliverables/` folder for a task whose blockers are already in place — typically:

```
blocker_registry.json / .md
modified_problem_statement.txt / modified_requirements.txt / modified_public_interfaces.txt
golden_patch_obstructed.diff / test_patch_obstructed.diff / relevant_tests.txt   (setup_patch.diff if Scenario 2)
Dockerfile / .hilbench_env   (plan.md / validate_obstructed_result.txt when present)
```

Plus an independent evaluation:

- **The 12 static model evals, run async** — `eval_test_list`, `eval_test_relevance`,
  `eval_golden_patch`, `eval_blocker_type`, `eval_blocker_independence`, `eval_blocker_objective`,
  `eval_blocker_critical_implementation`, `eval_blocker_descriptions`, `eval_blocker_questions`,
  `eval_blocker_distribution`, `eval_self_reference`, `eval_request`.
- **Maybe** the async full Harbor checks (`check1` guessable? / `check2` solvable-with-resolutions?).
  Don't depend on these being present.

---

## The one rule that defines this workflow

**Verify BOTH checks, and re-verify BOTH after every edit — the two checks pull against each
other.** A fix aimed at one routinely regresses the other:

| You edit to fix… | How you fix it | What it can silently regress |
|---|---|---|
| **Check 2** (`FAIL_UNSOLVABLE`) | add clarity/detail to the spec or resolution so agents can solve *with* resolutions | the added detail leaks the answer → a blocker becomes guessable → **Check 1 → `FAIL_GUESSABLE`** |
| **Check 1** (`FAIL_GUESSABLE`) | diverge the resolution to a non-obvious value; strip leakage from spec/identifiers | the task becomes unsolvable even *with* resolutions → **Check 2 → `FAIL_UNSOLVABLE`** |

There is no "fixing just Check 1" or "just Check 2." Every change is followed by re-verifying
**both**.

---

## 0. Provision the sandbox

You need the full sandbox (container + Harbor) to reproduce checks and re-verify fixes. In the
**chat window**:

```
/hilbench-provision
```

Then, in the **terminal**:

```bash
source deliverables/.hilbench_env
```

(See the Stage 0 notes in `CONTRIBUTOR_GUIDE.md` for image-source / auto-checkout behavior.)

---

## 0.5 Reproduce the baseline — and catch a dependency-broken image

Before triaging any flag, confirm the task actually *runs* on the provisioned image. Run the
dynamic gate on the artifacts as received:

```
/hilbench-validate-obstructed
```

This resets to the parent commit, bakes the setup patch, and runs the authored obstructed
test+golden through a real FAIL→PASS in the container. It's the one step that proves the
*received* task reproduces — a "verified, no changes" review otherwise never runs it.

**It also surfaces a missing-dependency image.** If the shipped image can't run the tests, the
verdict is `OBSTRUCTED_CHECK_ERROR` (or a misleading `OBSTRUCTED_F2P_FAIL`) — check
`deliverables/obstructed_after_stderr.log` for `ModuleNotFoundError` / `ImportError` /
`command not found`. (To confirm it's the environment and not the authored artifacts, run
`/hilbench-validate-original` in full-F2P mode: if the pristine original *also* errors, it's the
image; if it passes, the problem is in the artifacts.)

**Fixing a dependency-broken image.** The image the task ships with (from `task_info.txt`) is
missing a dependency needed to run the tests. Fix it with a corrected build-from-source
Dockerfile:

1. Author `deliverables/Dockerfile` — a real recipe (`FROM …` + the missing installs), using
   the fail-fast pattern (&&-chained installs with an `import <pkg>` sanity check; never a
   `;`-chained `|| true`, which caches an empty image).
2. Re-provision — `/hilbench-provision` now **prefers a buildable `deliverables/Dockerfile`
   over the `task_info` source** and builds from it (its content hash tags the image, so an
   edited Dockerfile always rebuilds). The built image is what every later stage and the grade
   run against.
3. Re-run `/hilbench-validate-obstructed` — it should now reach `OBSTRUCTED_PATCHES_OK`.

The corrected `deliverables/Dockerfile` must ship with the task (it *is* the task's image
source once present). If you can't produce a working Dockerfile, the task is broken — flag it
rather than submitting.

---

## 1. Triage the flagged issues

Start from the async results you were given — they already tell you what was flagged. You don't
need to re-run everything up front (that's the pre-submit sweep in §4). Here you just confirm and
understand each flag well enough to fix it.

Optionally reproduce a specific flag to confirm it before fixing:

```
/hilbench-run-eval <eval_name>     # re-judge ONE static eval, verbatim, read-only
```

To confirm a failing **check** and see *why* it failed, re-grade just that check — this prints the
diagnostic trajectory summaries + agent patches you need:

```
/hilbench-evaluate-check1          # authoritative single-check re-grade (heavy)
/hilbench-evaluate-check2
```

(For a cheap approximation while iterating, the `/hilbench-check1` / `/hilbench-check2` pre-screens
are faster but non-authoritative.)

**Watch the memorization trap on Check 1.** These are well-known OSS repos at publicly known
commits, so a resolution-free solver can reconstruct the *real* upstream behavior from memory —
"guessing" a blocker even though nothing in the task leaked the answer. If a blocker's correct
resolution equals the obvious/real upstream behavior, Check 1 will fail it. The fix is to make the
resolution a **non-standard, project-specific value that diverges from the real upstream default**,
with the tests asserting your value.

---

## 2. Where each failing signal lives (fix map)

| Failing item | Fix in | Note |
|---|---|---|
| `eval_request` | modified problem statement / requirements / interfaces | remove the leak/embedded solution |
| `eval_golden_patch` | `golden_patch_obstructed.diff` | missing impl, or an out-of-scope change |
| `eval_test_list` / `eval_test_relevance` | `test_patch_obstructed.diff`, `relevant_tests.txt` | tighten/add tests — never weaken |
| `eval_blocker_critical_implementation` (Test A) | golden / tests / the relevant resolution | resolution must be test-required |
| `eval_blocker_objective` / `_descriptions` / `_questions` / `_type` / `eval_self_reference` | `blocker_registry.json` | wording of one blocker |
| `eval_blocker_independence` | `blocker_registry.json` | coupled resolutions — reword, or regenerate the pair |
| `eval_blocker_distribution` | `blocker_registry.json` | wrong type counts — often a regenerate |
| **Check 1 `FAIL_GUESSABLE`** | `blocker_registry.json` resolution + modified spec | diverge to a non-obvious value; strip leakage — then re-check Check 2 |
| **Check 2 `FAIL_UNSOLVABLE`** | modified spec / resolution / golden | add minimal missing detail; regenerate golden — then re-check Check 1 |

---

## 3. Fix it

Pick the smallest fix that works, and follow the no-regression rules in
[`references/optional-repair.md`](./references/optional-repair.md) (smallest change; never weaken
tests; no new leaks/self-reference). You have three options:

**a) Improve in place** — for a localized issue (one blocker's wording, a missing spec detail, a
resolution that must diverge from the upstream default). Edit the canonical files directly —
`blocker_registry.json`, `modified_*`, and the obstructed patches — since every tool reads those
exact filenames. The pristine original is preserved upstream, so no local backup is needed. Then
re-run the gates (below) — start at `/hilbench-validate-artifacts` for an artifact/patch-only edit,
or at `/hilbench-validate-registry` if you touched `blocker_registry.json`.

**b) Regenerate specific blockers** — when a few blockers are weak/guessable but the rest are fine:

```
/hilbench-registry-regenerate <id[,id...]>
```

It re-does **only** the named blockers (kept independent, non-overlapping, and type-locked so the
distribution holds) and merges them back. It does **not** touch the specs/patches — those are
*derived* from the registry, so you re-derive them with the gate stream below.

**c) Regenerate the whole task** — when problems are systemic (several guessable/coupled blockers,
or a wrong distribution): `/hilbench-registry-generate` (destructive — overwrites the entire
registry), then run the same gate stream below.

### Re-run the gates after every fix — this is mandatory

**You MUST re-run the pipeline gates after any fix — whether you regenerated blockers, edited a
blocker in place, or only hand-fixed a patch. A fix is not done until the gates pass again.**
*Where you start depends on what you changed*, because `/hilbench-inject` re-derives **all** specs
+ obstructed patches from the registry and **overwrites any hand edits to those files**:

| What you changed | Re-run, in order | Note |
|---|---|---|
| **Registry / blockers** — regenerated (b/c), or edited a resolution / description / type / distribution | `/hilbench-validate-registry` → `/hilbench-inject` → `/hilbench-validate-artifacts` → `/hilbench-validate-obstructed` | inject re-derives everything from the registry — run it **before** any artifact-only hand-tuning, never the reverse |
| **Only artifacts** — obstructed patches and/or modified specs, registry untouched | `/hilbench-validate-artifacts` → `/hilbench-validate-obstructed` | do **NOT** run `/hilbench-inject` — it would re-derive from the registry and wipe your patch/spec fix |

The full stream, when the registry changed:

```
/hilbench-validate-registry     # confirm the merged registry + full set
/hilbench-inject                # re-derive ALL modified specs + obstructed patches from the registry
/hilbench-validate-artifacts    # alignment map + patch-dependent evals
/hilbench-validate-obstructed   # dynamic FAIL→PASS on the re-authored patches
```

> **Fixing only the patches? Do NOT start at `/hilbench-validate-registry`, and do NOT run
> `/hilbench-inject`.** The registry is unchanged and already green, so those steps have nothing
> new to do — and `/hilbench-inject` would overwrite your patch fix by re-deriving from the
> registry. Start at `/hilbench-validate-artifacts`, then `/hilbench-validate-obstructed`. Start at
> `/hilbench-validate-registry` only when you actually changed the registry/blockers.

This gate stream only re-derives/re-validates the artifacts — it is **not** the full submit sweep.
Once it's clean, still run the §4 loop (all 12 evals + BOTH checks) before submitting.

---

## 4. Verify, then submit

After any fix, run this loop until everything is clean at once:

1. **Re-run all 12 evals** — `/hilbench-run-evals`. Always run the full set, not just the one you
   touched: fixing one eval can regress another (e.g., tightening a resolution for
   `eval_blocker_objective` can trip `eval_blocker_independence` or `eval_request`).
2. **Re-run BOTH checks** — not just the one you fixed (see the rule above). Iterate with the fast
   pre-screens `/hilbench-check1` / `/hilbench-check2`.
3. **Fix and repeat** — for anything still failing (or newly regressed), go back to §3, fix it,
   then re-run steps 1–2. Repeat until all 12 evals and both checks pass together.
4. **Final sign-off:** `/hilbench-evaluate-full --fresh` — a full authoritative Harbor grade with
   no incremental skips.

Submit only when the 12 evals and both checks pass locally. Nothing auto-advances; you decide when
it's clean.

---

## 5. Track what you changed

Keep a running record of every change and why — you'll submit a brief summary. A change is anything
you edited from what you received (a blocker resolution/description, a spec file, a patch, the
distribution, or a decision to regenerate). A few lines each:

```
- blocker_registry.json (blocker_b): diverged resolution from the real upstream default (30s)
  to a project-specific 45s — Check 1 was FAIL_GUESSABLE (memorization trap).
- modified_requirements.txt: added the missing pagination-limit sentence — Check 2 was
  FAIL_UNSOLVABLE; re-checked Check 1 after, still clean.
- regenerated: 3 of 4 blockers were guessable, hand-fixing wasn't worth it.
```

If nothing needed changing, say exactly that — "verified, no changes."

## Quick reference

```
/hilbench-provision                 # 0  full sandbox (container + Harbor)
source deliverables/.hilbench_env   #    terminal
/hilbench-validate-obstructed       # 0.5 reproduce received task; also catches a dep-broken image
#     dep-broken image? author deliverables/Dockerfile (FROM + missing installs), then:
/hilbench-provision                 #     re-provision — prefers a buildable deliverables/Dockerfile
/hilbench-validate-obstructed       #     re-verify -> OBSTRUCTED_PATCHES_OK
# 1  triage the async results; reproduce a flag if useful:
/hilbench-run-eval <name>           #     re-judge one static eval
/hilbench-evaluate-check1|2         #     re-grade one check (trajectories on failure)
# 3  fix it: improve in place  |  regenerate blocker(s)  |  regenerate whole task
/hilbench-registry-regenerate <ids> #     regenerate named blockers, then re-derive (below)
# registry/blocker change -> full stream (inject re-derives + OVERWRITES artifacts):
/hilbench-validate-registry ; /hilbench-inject ; /hilbench-validate-artifacts ; /hilbench-validate-obstructed
# artifact/patch-only fix (registry unchanged) -> SKIP validate-registry + inject:
/hilbench-validate-artifacts ; /hilbench-validate-obstructed
# 4  verify, then submit:
/hilbench-run-evals                 #     all 12 evals (fixing one can regress another)
/hilbench-check1 ; /hilbench-check2 #     re-verify BOTH checks after EVERY edit
/hilbench-evaluate-full --fresh     #     final sign-off; submit only on a clean local pass
# 5  log each change + why ("verified, no changes" if none)
```
